Prompt Implementación AIRA

Estoy trabajando en el frontend de un dashboard IoT para monitoreo de calidad del aire. Ya existe un proyecto funcional en HTML, CSS y JavaScript estático, compatible con GitHub Pages. Necesito que implementes la integración visual y funcional de AIRA, el módulo de Inteligencia Artificial del sistema.

Contexto técnico del proyecto:

El sistema completo ya funciona así:

ESP32 → MQTT → Raspberry Pi → SQLite local + Supabase → Dashboard web en GitHub Pages

La ESP32 mide:
- PM1.0
- PM2.5
- PM10
- NH3 estimado
- Temperatura
- Humedad
- Presión
- Estado ambiental: Normal / Precaución / Peligro
- Causa principal del estado
- muted

La Raspberry Pi:
- Corre Mosquitto.
- Recibe datos MQTT desde la ESP32.
- Guarda datos en SQLite.
- Sube los datos a Supabase.
- Ejecuta gateway_local.py automáticamente con systemd.
- Integra Gemini API desde backend, no desde frontend.
- La API key de Gemini está solo en la Raspberry.
- La secret key de Supabase está solo en la Raspberry.
- El frontend usa únicamente la Publishable Key de Supabase.

El dashboard ya tiene:
- index.html
- style.css
- app.js
- Supabase JS Client
- Chart.js
- Tarjetas principales
- Gráfica histórica
- Tabla de lecturas
- Filtros avanzados
- Polling cada 3 segundos
- Alertas de voz cuando el estado es Peligro

Ahora necesito agregar la sección de AIRA.

AIRA significa:

AIRA — Asistente Inteligente de Recomendación Ambiental

AIRA es una IA contextualizada para autoridades locales de Chía, Cundinamarca, en una zona periurbana de Sabana Centro con influencia de vías locales, áreas residenciales y actividad agropecuaria cercana.

AIRA no debe estar activa todo el tiempo porque consume tokens. La lógica backend ya está pensada para que AIRA se active:
1. Automáticamente cuando el estado sea Peligro.
2. Manualmente cuando el usuario presione un botón en el dashboard.

El frontend NO debe llamar directamente a Gemini.
El frontend NO debe usar ninguna API key de Gemini.
El frontend NO debe usar secret key de Supabase.
El frontend solo debe:
1. Leer recomendaciones desde la tabla ai_recommendations.
2. Crear solicitudes manuales insertando registros en la tabla ai_requests.

Tablas de Supabase ya creadas o previstas:

Tabla public.ai_recommendations:
- id
- timestamp_utc
- trigger_source
- estado
- causa
- recommendation
- model
- input_json
- created_at

Tabla public.ai_requests:
- id
- request_type
- status
- source
- created_at
- processed_at
- error

Políticas esperadas:
- ai_recommendations permite SELECT a anon.
- ai_requests permite INSERT a anon solo con request_type='manual' y status='pending'.
- El dashboard no debe actualizar ni borrar ai_requests.

Archivos que debes modificar:
- index.html
- style.css
- app.js

No modificar:
- esp/prueba.ino
- raspberry/gateway_local.py
- archivos de configuración secretos
- no agregar backend
- no usar frameworks pesados
- mantener compatibilidad con GitHub Pages

FLUJO FUNCIONAL ESPERADO:

1. El usuario abre el dashboard.
2. El dashboard carga lecturas desde Supabase.
