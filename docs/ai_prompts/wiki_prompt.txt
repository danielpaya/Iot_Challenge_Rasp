Prompt utilizado en claude

Estoy trabajando en un proyecto IoT de monitoreo de calidad del aire para el Challenge #3 del curso Internet de las Cosas.

Contexto general del sistema ya implementado:

El sistema tiene esta arquitectura funcional:

ESP32 → MQTT → Raspberry Pi → SQLite local + Supabase → Dashboard web en GitHub Pages

Estado actual del proyecto:

1. La ESP32 ya mide y procesa datos ambientales:
   - PM1.0
   - PM2.5
   - PM10
   - NH3 estimado
   - Temperatura
   - Humedad
   - Presión
   - Estado ambiental: Normal / Precaución / Peligro
   - Causa principal del estado
   - Estado de silenciamiento del buzzer

2. La ESP32 publica los datos mediante MQTT al tópico:
   sabana/aire/nodo1

3. La Raspberry Pi funciona como Gateway IoT:
   - Corre Mosquitto como broker MQTT.
   - Ejecuta un script Python llamado gateway_local.py.
   - Recibe los datos MQTT desde la ESP32.
   - Guarda cada lectura en SQLite local.
   - Sube cada lectura a Supabase.
   - El servicio corre automáticamente con systemd al prender la Raspberry.

4. Supabase ya tiene una tabla llamada readings con esta estructura aproximada:
   - id
   - timestamp_utc
   - pm1
   - pm25
   - pm10
   - nh3
   - temperature
   - humidity
   - pressure
   - estado
   - causa
   - muted
   - raw_json
   - created_at

5. Ya existe un dashboard web funcional en localhost que lee Supabase usando JavaScript desde el navegador.
   El dashboard está hecho con:
   - index.html
   - style.css
   - app.js
   - Supabase JS Client
   - Chart.js

6. El dashboard actual:
   - Lee la tabla readings desde Supabase.
   - Muestra tarjetas con los últimos valores.
   - Muestra el estado ambiental.
   - Muestra causa principal.
   - Muestra una gráfica histórica de PM2.5 y PM10.
   - Muestra una tabla con las últimas lecturas.
   - Actualiza los datos mediante polling cada 3 segundos.
   - Está pensado para publicarse en GitHub Pages.

OBJETIVO 1: Filtros avanzados para la tabla de datos

Actualmente la tabla muestra las últimas lecturas. Quiero agregar una sección de filtros para que el usuario pueda analizar mejor la información histórica.

Implementa filtros por:

1. Fecha específica:
   - Un input tipo date.
   - Debe filtrar registros del día seleccionado.
   - Debe comparar contra timestamp_utc.

2. Rango de fechas:
   - Fecha inicial.
   - Fecha final.
   - Debe traer registros entre ambas fechas.

3. Rango de horas:
   - Hora inicial.
   - Hora final.
   - Debe permitir filtrar registros dentro de un intervalo horario.
   - Ejemplo: de 08:00 a 12:00.

4. Estado ambiental:
   - Select con opciones:
     - Todos
     - Normal
     - Precaución
     - Peligro

5. Tipo de variable o dato:
   - Select con opciones:
     - Todos
     - PM1.0
     - PM2.5
     - PM10
     - NH3
     - Temperatura
     - Humedad
     - Presión
   - Este filtro no debe ocultar filas necesariamente, sino permitir resaltar o priorizar la variable seleccionada en la tabla y/o gráfica.
   - Si se selecciona una variable específica, la gráfica debe mostrar esa variable principal.
   - Si se selecciona "Todos", la gráfica puede mostrar PM2.5 y PM10 como actualmente.

6. Botón "Aplicar filtros":
   - Debe ejecutar la consulta a Supabase usando los filtros seleccionados.

7. Botón "Limpiar filtros":
   - Debe restaurar la vista normal de últimos registros.
   - Debe volver al polling automático cada 3 segundos.

8. Mientras hay filtros activos:
   - El polling automático puede pausarse para no sobrescribir los resultados filtrados.
   - Debe mostrarse un indicador visual tipo "Filtros activos".
   - El usuario debe poder limpiar filtros para volver al modo en vivo.

9. La tabla debe seguir mostrando columnas:
   - Hora / fecha
   - PM1
   - PM2.5
   - PM10
   - NH3
   - Temperatura
   - Humedad
   - Presión
   - Estado
   - Causa

10. Agrega una indicación de cuántos registros se están mostrando:
   - Ejemplo: "Mostrando 40 registros"
   - O "Mostrando 18 registros filtrados"

11. La consulta a Supabase debe estar optimizada:
   - Usar select("*")
   - Usar order("timestamp_utc", { ascending: false })
   - Usar limit razonable, por ejemplo 100 o 200 registros para filtros.
   - Aplicar gte/lte cuando haya rango de fechas.
   - Aplicar eq cuando haya estado específico.

OBJETIVO 2: Alerta por voz cuando el estado sea Peligro

Quiero que el dashboard emita una alerta audible por voz cuando llegue un dato con estado "Peligro".

Implementa esto con la Web Speech API del navegador, usando speechSynthesis.

Requisitos:

1. Cuando el último registro recibido tenga estado "Peligro", el navegador debe decir por voz algo como:
   "Alerta de peligro. Se detectaron niveles críticos de calidad del aire. Revise el dashboard y active el protocolo de respuesta."

2. La voz no debe repetirse cada 3 segundos si el sistema sigue en peligro.
   Debe evitar spam de voz.
   Implementa una bandera como lastSpokenDangerId o dangerAlertAlreadySpoken.

3. La alerta debe activarse solamente cuando:
   - Entra un nuevo registro con estado "Peligro", o
   - El estado cambia de Normal/Precaución a Peligro.

4. Si el estado vuelve a Normal o Precaución, debe resetearse la bandera para que una futura condición de Peligro vuelva a generar voz.

5. Agrega un control visual en el dashboard:
   - Botón para activar/desactivar alertas de voz.
   - Texto de estado: "Alertas de voz activadas" o "Alertas de voz desactivadas".

6. Por restricciones de navegadores, la voz puede requerir interacción previa del usuario.
   Para manejar esto:
   - Agrega un botón "Activar alertas de voz".
   - Al hacer clic, se habilita la variable voiceAlertsEnabled = true.
   - Opcionalmente, hacer una prueba de voz corta:
     "Alertas de voz activadas."

7. La alerta por voz debe funcionar en el frontend sin backend.

8. Si speechSynthesis no está disponible, mostrar un mensaje discreto:
   "El navegador no soporta alertas de voz."

9. Además de voz, cuando estado sea Peligro:
   - La tarjeta de estado ambiental debe cambiar visualmente a un estilo crítico.
   - Puede agregarse una clase CSS especial.
   - Puede mostrar texto tipo "ALERTA CRÍTICA".
